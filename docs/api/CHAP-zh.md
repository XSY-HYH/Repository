# CHAP 协议与锯齿交互模型（ZIM）

> **注意：本协议并非旧版的 Challenge-Handshake Authentication Protocol（挑战握手认证协议）。** 这是一个名称相同但完全不同的协议，全称为 Chain Hash Authentication Protocol（链式哈希认证协议）。

---

## 一、协议定位

CHAP（Chain Hash Authentication Protocol）是一个通用协议，理论上可以适配 HTTP、HTTPS、TCP、WebSocket 等多种传输协议。

CHAP 的核心设计目标是连接状态管理，而非多用户认证。用户数量增加会直接推高服务端的解密成本，因此该协议不适合大规模多用户场景。

CHAP 具备可扩展性。最初的版本确实不适用于多用户，但其衍生版本（如 CHAP-IEM，即 ID 加密模式）在理论上可以适配更多样的使用场景。

---

## 二、核心流程

![CHAP 流程图](./chap-zh.png)

客户端需要与服务端预先持有相同的密钥。整个交互流程如下：

**登录阶段**

1. 客户端填写用户名和密钥，将密钥转换为哈希值
2. 使用该哈希值对用户名进行 AES 加密
3. 将加密后的数据包发送给服务端

**服务端验证**

4. 服务端使用预配置的密钥解密数据包
5. 若解密失败或用户名不匹配，直接拒绝连接
6. 验证成功后，服务端返回操作结果为 OK，并附带一个 ID

**ID 链式管理**

7. 客户端后续每次操作都必须携带当前持有的 ID
8. 服务端收到操作包后，使用密钥解密并匹配操作 ID
9. 匹配成功后，服务端销毁当前 ID，同时生成一个新的 ID
10. 服务端将新 ID 与操作结果打包加密，返回给客户端
11. 客户端解密获得新 ID，用于下一次操作

**示例**：假设登录成功返回的 ID 是 1，客户端下一次数据包就必须使用密钥加密，同时包含操作 ID=1 以及操作数据。服务端解密并匹配 ID=1 后，会销毁 ID 1，生成 ID 2，再把 ID 2 和结果打包加密返回给客户端。后续操作依此类推。

---

## 三、异常恢复机制

当响应包丢失导致客户端 ID 与服务端不同步时，协议内置了自动恢复逻辑：

1. 客户端使用旧 ID 发送操作包
2. 服务端解密成功，但发现该 ID 已失效
3. 服务端返回恢复包，包含当前有效 ID 及同步指令
4. 客户端更新本地 ID 后发送确认包
5. 服务端确认后，双方恢复正常通信

若采用 CHAP-IEM 变体（ID 直接作为加密密钥），密钥不同步时则需要客户端重新登录以重建密钥链。

---

## 四、安全特性

攻击者无法在缺少密钥的情况下解密截获的密文，也无法读取其中的 ID。即使攻击者重放旧的数据包，由于 ID 已被销毁，服务端会直接拒绝该操作。

---

## 五、深层本质：锯齿交互模型（ZIM）

CHAP 代表的并非一个专有框架，其更深层次的本质是**锯齿交互模型（Zigzag Interaction Model，简称 ZIM）**。

该模型的核心特征是：客户端与服务端在交互过程中，两次会话之间永远相差一个齿，但整体始终保持啮合状态。每一次请求携带当前齿位，每一次响应推进到下一个齿位，形成持续的链式状态迁移。

CHAP 只是 ZIM 模型的一个具体实现典范。任何遵循该模型的协议，都可以视为 CHAP 家族的成员。

---

## 六、加密算法推荐

针对 ZIM 协议实现：

### 推荐方案：AES-256-CBC + Encrypt-then-MAC（HMAC-SHA-256）

- IV 可使用随机数或计数器（复用不会造成灾难性后果）
- MAC 提供完整性保护，可防止填充预言机攻击

### 替代方案：AES-256-GCM（如硬件支持加速）

- **警告**：Nonce 复用会造成灾难性后果。实现必须在同一密钥下的每次加密操作中使用唯一的 Nonce。
- 在 CHAP 模式（固定密钥 K）下，这需要跨会话维护持久状态。
- 不推荐用于缺乏可靠熵源的嵌入式环境。

### 最小方案（资源受限场景）：仅使用 AES-256-CBC

- 依赖 ZIM 的 ID 链进行完整性验证
- 实现必须对所有失败情况返回相同的错误响应
- 可接受用于内部网络或单用户场景
