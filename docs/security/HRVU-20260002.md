# HRVU-20260002 - Logs API Sensitive Information Leak

## Number
**HRVU-20260002**

- **HRVU**: High-Risk Vulnerability Update
- **202600**: Source Update ID (Unknown source/Proactive discovery)
- **02**: Second issue after this update

## Vulnerability Overview

### Discovery Date
April 6, 2026

### Vulnerability Type
Information Leak - Sensitive Logs Public Access

### Risk Level
**High**

### Affected Scope
`/api/logs` and `/api/logs/{logFile}` endpoints

## Vulnerability Details

### Problem Description
The server's log endpoints `/api/logs` and `/api/logs/{logFile}` are publicly accessible to all users without authentication. Log files may contain sensitive information such as:

- Client IP addresses
- Access paths and request parameters
- Admin operation records
- System configuration information
- Error stack traces
- Potential sensitive file paths

### Attack Scenarios

#### 1. Information Gathering
```
Attacker accesses:
GET /api/logs

Gets list of all log files, understanding system running status
```

#### 2. Sensitive Information Acquisition
```
Attacker accesses:
GET /api/logs/2026-04-06.log

Gets detailed log content, including:
- Admin login time and IP
- File access records
- System error information
- Configuration change records
```

#### 3. User Behavior Tracking
```
By analyzing logs, attackers can:
- Track specific user access patterns
- Discover hidden directory structures
- Identify system weaknesses
```

### Potential Harm
- Privacy leak: User IPs and access records exposed
- System information leak: Provides intelligence for further attacks
- Admin information exposed: May be used for targeted attacks
- Business information leak: File access patterns may expose business logic

## Fix Solution

### Implementation Method
1. Remove public `/api/logs` endpoint
2. Move logs functionality to admin WebSocket panel
3. Require admin authentication for access

### Endpoint Changes

#### Before Fix
```
GET /api/logs              → Returns log file list (public)
GET /api/logs/{logFile}    → Returns log file content (public)
```

#### After Fix
```
GET /api/logs              → 404 Not Found
GET /api/logs/{logFile}    → 404 Not Found
GET /admin/api/logs        → 404 Not Found (removed)
WebSocket admin panel      → get_logs / get_log operations (requires admin auth)
```

### Access Control
- Logs accessed via WebSocket admin panel
- Requires `AdminEnabled = true`
- Requires CHAP authentication

## Verification Method

### Test Public Endpoints (Should Return 404)
```bash
curl -I https://your-server:port/api/logs
# Expected: HTTP/2 404

curl -I https://your-server:port/api/logs/2026-04-06.log
# Expected: HTTP/2 404

curl -I https://your-server:port/admin/api/logs
# Expected: HTTP/2 404
```

## File Changes

### Modified Files
- `Controllers/SharedController.cs` - Remove public logs endpoint
- `Controllers/AdminController.cs` - Add admin logs endpoint

## Security Recommendations

### Immediate Actions
1. Ensure fixed version is deployed
2. Check logs for abnormal access records
3. Assess risk of leaked information

### Long-term Improvements
1. Log desensitization: Remove or encrypt sensitive information
2. Log rotation: Regularly clean old logs
3. Access auditing: Record log access behavior
4. Log classification: Separate sensitive and non-sensitive logs
