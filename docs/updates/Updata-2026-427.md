# Update Log - April 27, 2026

## Bug Fixes

### 1. WebSocket Connection Rejected by Header Filtering
- Fixed issue where WebSocket connections to `/admin/ws` were rejected due to missing `Accept` header
- Browser WebSocket API does not support custom headers during connection handshake
- **Solution**: Skip header filtering for WebSocket endpoints (`/admin/ws`, `*/ws`)
- Modified `RequestHeaderFilteringService.ValidateRequestHeaders()` to detect WebSocket paths

### 2. Temporary Block List Format Refactoring
- Changed `TemporarilyBlockedIPs` format from `"IP,timestamp|IP,timestamp"` to `"IP1,IP2,IP3"` (comma-separated IPs only)
- Timestamps are now managed in memory via `_tempBlockTimes` ConcurrentDictionary
- **Reason**: Cleaner config format, timestamps are runtime data not configuration

### 3. Restart Resets Temporary Ban Timer
- On server restart, all temporarily blocked IPs get their block time reset to current time
- This ensures consistent behavior: restart = fresh timer for all temp bans
- Implemented via `InitializeTempBlockTimes()` method on service startup

## Feature Improvements

### 1. Permanent Ban Escalation Mechanism
- When a temporarily blocked IP continues to exceed `MaximumRequestsPerSecond` threshold:
  - IP is immediately permanently banned (added to `BlockedIPs`)
  - IP is removed from `TemporarilyBlockedIPs`
  - Both memory cache and config file are updated
- New method `RemoveFromTemporaryBlockList()` handles cleanup
- Added logging for escalation events (`throttling.removed_from_temporary`)

### 2. Request Throttling Service Rewrite
- Complete refactoring of temporary block management:
  - `_tempBlockTimes`: In-memory dictionary tracking IP → block start time
  - `IsInTemporaryBlockList()`: Checks against in-memory timestamps with configurable duration
  - `AddToTemporaryBlockList()`: Writes only IP to config, timestamp to memory
  - `RemoveFromTemporaryBlockList()`: Cleans up both memory and config
  - `CleanupExpiredData()`: Timer-based cleanup removes expired entries from both stores

## Feature Upgrade: CHAP → CHAP-IEM (ID Encryption Mode)

### Protocol Overview
CHAP-IEM is a derivative variant of the standard CHAP protocol. The core difference:
- **Standard CHAP**: Always uses pre-shared key K for encryption
- **CHAP-IEM**: Login phase uses K for authentication, then switches to **ID chain encryption mode** — encryption key auto-rotates per operation

### Workflow
```
Login Phase (same as standard CHAP):
  Client ──[K-encrypt username]──→ Server verifies identity
  Server ──[K-encrypt response+ID_1]──→ Client obtains ID_1

Normal Operations (ID chain mode):
  Op1: Client[ID_1-encrypt]──→ Server[ID_1-decrypt]──→ Execute, generate ID_2
       Server ──[ID_1-encrypt result+ID_2]──→ Client gets ID_2, key rotates to ID_2

  Op2: Client[ID_2-encrypt]──→ Server[ID_2-decrypt]──→ Execute, generate ID_3
       Server ──[ID_2-encrypt result+ID_3]──→ Client gets ID_3, key rotates to ID_3

  Key Chain: K → ID_1 → ID_2 → ID_3 → ...
```

### Exception Recovery Mechanism
When client and server ID chains get out of sync (e.g., packet loss):
1. Server detects decryption failure, attempts K-based parsing (recovery channel)
2. If successful, encrypts recovery packet using K (containing current valid ID)
3. Client decrypts recovery packet using K, obtains current ID, resynchronizes
4. **Security**: K used only as recovery channel, not for normal operations; forward secrecy preserved

### Backend Changes (ChapAuthService.cs)

| Aspect | Standard CHAP | CHAP-IEM |
|--------|---------------|----------|
| Encryption key | Always K | Switches to ID chain after login |
| Key storage | `_sessionIdToKey` (password hash) | `_sessionCurrentKeys` (per-session ID keys) |
| Operation decrypt | Direct decrypt with K | Iterate all session current ID keys to match |
| Response encrypt | Encrypt with K | Encrypt with **old ID** (carrying new ID) |
| Recovery mechanism | None | K as recovery channel |

**New Methods:**
- `GenerateId()` / `GenerateIdBytes()` — Generate 32-byte random ID
- `HandleRecoveryRequest()` — Proactively send recovery packet
- `GetK()` — Lazy-fetch pre-shared key

### Controller Changes (AdminController.cs)
- After login success: `sessionKey = Convert.FromBase64String(response.NewId)` (switch from K to ID_1)
- Operation response: Use `encryptKey` (old ID) instead of new ID for encryption
- After each successful operation: `sessionKey` auto-rotates to new ID

### Frontend Changes (Admin.JS)
- **Variable split**: `k` (pre-shared key, login & recovery only) + `currentKey` (current ID key, operation encrypt/decrypt)
- **New utility**: `base64ToBytes()` — Base64 string to Uint8Array conversion
- **sendOperation**: Uses `currentKey` for request encryption/response decryption, auto-rotates on NewId receipt

### Security Improvements
| Feature | Standard CHAP | CHAP-IEM |
|---------|---------------|----------|
| Key fixity | K never changes | Auto-rotates per operation |
| Single-packet leak impact | Can decrypt all subsequent comms | Affects only current single operation |
| Forward secrecy | Not supported (K leak = total compromise) | Supported (old keys cannot derive new keys) |
