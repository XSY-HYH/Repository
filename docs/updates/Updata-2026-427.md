# Update Log - April 27, 2026

## Bug Fixes

### 1. WebSocket Connection Rejected by Header Filtering
- Fixed issue where WebSocket connections to `/admin/ws` were rejected due to missing `Accept` header
- Browser WebSocket API does not support custom headers during connection handshake
- **Solution**: Skip header filtering for WebSocket endpoints (`/admin/ws`, `*/ws`)
- Modified `RequestHeaderFilteringService.ValidateRequestHeaders()` to detect WebSocket paths

### 2. Temporary Block List Format Refactoring
- Changed `TemporarilyBlockedIPs` format from `"IP,timestamp|IP,timestamp"` to `"IP1,IP2,IP3"` (comma-separated IPs only)
- Timestamps are now managed in memory via `_tempBlockTimes` ConcurrentDictionary
- **Reason**: Cleaner config format, timestamps are runtime data not configuration

### 3. Restart Resets Temporary Ban Timer
- On server restart, all temporarily blocked IPs get their block time reset to current time
- This ensures consistent behavior: restart = fresh timer for all temp bans
- Implemented via `InitializeTempBlockTimes()` method on service startup

## Feature Improvements

### 1. Permanent Ban Escalation Mechanism
- When a temporarily blocked IP continues to exceed `MaximumRequestsPerSecond` threshold:
  - IP is immediately permanently banned (added to `BlockedIPs`)
  - IP is removed from `TemporarilyBlockedIPs`
  - Both memory cache and config file are updated
- New method `RemoveFromTemporaryBlockList()` handles cleanup
- Added logging for escalation events (`throttling.removed_from_temporary`)

### 2. Request Throttling Service Rewrite
- Complete refactoring of temporary block management:
  - `_tempBlockTimes`: In-memory dictionary tracking IP → block start time
  - `IsInTemporaryBlockList()`: Checks against in-memory timestamps with configurable duration
  - `AddToTemporaryBlockList()`: Writes only IP to config, timestamp to memory
  - `RemoveFromTemporaryBlockList()`: Cleans up both memory and config
  - `CleanupExpiredData()`: Timer-based cleanup removes expired entries from both stores

## Technical Details

### Temporary Ban Flow (New)
```
Request → Check rate → ≥ AlarmThreshold? 
    ↓ Yes              ↓ No
Add to temp list      Allow request
(Reset timer on each hit within duration)

While temp-blocked, if requests continue:
    ↓ Exceed MaxThreshold?
Yes → Permanently ban + Remove from temp list
No  → Continue blocking (timer counting down)
```

### Memory vs Config Separation
| Data | Storage | Persistence |
|------|---------|-------------|
| IP address | Config file (`TemporarilyBlockedIPs`) | Survives restart |
| Block start time | Memory (`_tempBlockTimes`) | Reset on restart |
| Block duration | Config (`BlockDurationMinutes`) | Configurable |
